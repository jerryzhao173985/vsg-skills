# Patterns

These are the copy-paste recipes for building a working VulkanSceneGraph (VSG) application. Every VSG object is reference counted and created through the static `ClassName::create(...)` factory generated by the `Inherit<>` CRTP base (core/Inherit.h:35), so you never write raw `new` — `create` returns a `ref_ptr<>` smart pointer (core/ref_ptr.h:21) that owns the object. Pull in the whole public API with `#include <vsg/all.h>`, or include the specific headers cited per recipe. The recipes below build up from a minimal window to scene construction, interaction, model loading, custom event handling, the frame loop, and finally multithreaded recording.

## Minimal application skeleton

A VSG app wires together a `Viewer`, a `Window` (created from a `WindowTraits`), a scene root, a `Camera` (a `ProjectionMatrix` + a `ViewMatrix`), a `View` (camera + scene), a `RenderGraph` (the render pass), and a `CommandGraph` (records the subgraph into a Vulkan command buffer). The command graph is handed to the viewer via `assignRecordAndSubmitTaskAndPresentation`, then `compile()` builds the Vulkan resources before the loop runs. Note the convenience function `createCommandGraphForView` (app/CommandGraph.h:66) collapses the View/RenderGraph/CommandGraph wiring into one call — used here to keep the skeleton minimal.

```cpp
#include <vsg/all.h>

int main()
{
    auto traits = vsg::WindowTraits::create("vsg viewer");           // app/WindowTraits.h:33
    traits->width = 1280;                                            // app/WindowTraits.h:49
    traits->height = 1024;                                           // app/WindowTraits.h:50

    auto viewer = vsg::Viewer::create();                             // app/Viewer.h:32,35
    auto window = vsg::Window::create(traits);                       // app/Window.h:37
    viewer->addWindow(window);                                       // app/Viewer.h:41

    auto scene = vsg::Group::create();                               // nodes/Group.h:23

    // camera: perspective projection + lookAt view, sized to the window
    auto perspective = vsg::Perspective::create(                     // app/ProjectionMatrix.h:49,69
        30.0, 1.6, 0.1, 1000.0);
    auto lookAt = vsg::LookAt::create(                               // app/ViewMatrix.h:53,71
        vsg::dvec3(0.0, -5.0, 0.0), vsg::dvec3(0.0, 0.0, 0.0), vsg::dvec3(0.0, 0.0, 1.0));
    auto camera = vsg::Camera::create(perspective, lookAt);          // app/Camera.h:26,31

    // wire View -> RenderGraph -> CommandGraph in one convenience call
    auto commandGraph = vsg::createCommandGraphForView(              // app/CommandGraph.h:66
        window, camera, scene);
    viewer->assignRecordAndSubmitTaskAndPresentation({commandGraph});// app/Viewer.h:124

    viewer->compile();                                               // app/Viewer.h:107

    while (viewer->advanceToNextFrame())                             // app/Viewer.h:102
    {
        viewer->handleEvents();                                      // app/Viewer.h:105
        viewer->update();                                            // app/Viewer.h:135
        viewer->recordAndSubmit();                                   // app/Viewer.h:137
        viewer->present();                                           // app/Viewer.h:139
    }
    return 0;
}
```

If you prefer to build the graph by hand rather than with the convenience function, construct a `vsg::View::create(camera, scene)` (app/View.h:43), wrap it in `vsg::RenderGraph::create(window, view)` (app/RenderGraph.h:34), and add that to a `vsg::CommandGraph::create(window)` (app/CommandGraph.h:32) via `addChild` (nodes/Group.h:39); `createRenderGraphForView` (app/RenderGraph.h:88) is the View+RenderGraph half of the helper.

## Building a scene graph

The scene is a tree of `Node`s. `Group` holds children added with `addChild` (nodes/Group.h:39). To position a subgraph, decorate it with a `MatrixTransform` whose `matrix` field (nodes/MatrixTransform.h:30) holds a `dmat4`; the transform is composed onto the modelview stack during recording (nodes/MatrixTransform.h:20-22). Geometry that needs its own pipeline/descriptors is grouped under a `StateGroup` whose state is added with `add` (nodes/StateGroup.h:48). For quick primitives, `vsg::Builder` produces ready-to-render subgraphs.

```cpp
auto scene = vsg::Group::create();                                  // nodes/Group.h:23

// place a subgraph 2 units along X using a MatrixTransform
auto transform = vsg::MatrixTransform::create();                    // nodes/MatrixTransform.h:23,26
transform->matrix = vsg::translate(2.0, 0.0, 0.0);                  // nodes/MatrixTransform.h:30

// generate a box with the Builder
auto builder = vsg::Builder::create();                              // utils/Builder.h:109,112
vsg::GeometryInfo geom;                                             // utils/Builder.h:53
geom.position = vsg::vec3(0.0f, 0.0f, 0.0f);                        // utils/Builder.h:63
geom.color = vsg::vec4(1.0f, 0.0f, 0.0f, 1.0f);                    // utils/Builder.h:67
vsg::StateInfo state;                                               // utils/Builder.h:25
auto box = builder->createBox(geom, state);                        // utils/Builder.h:123

transform->addChild(box);                                           // nodes/Group.h:39 (inherited)
scene->addChild(transform);                                         // nodes/Group.h:39
```

`Builder` also offers `createSphere`, `createCylinder`, `createCone`, `createCapsule`, `createDisk`, `createQuad`, and `createHeightField` with the same `(GeometryInfo, StateInfo)` signature (utils/Builder.h:124-130). `StateGroup::create()` (nodes/StateGroup.h:31) is used when you assemble pipelines manually instead of via the Builder.

## Camera & interaction

A `Camera` pairs a projection with a view matrix (app/Camera.h:31). Use `Perspective(fov, aspectRatio, near, far)` (app/ProjectionMatrix.h:69) for the projection and `LookAt(eye, center, up)` (app/ViewMatrix.h:71) for the view. To let the user orbit/pan/zoom with mouse and touch, construct a `vsg::Trackball` bound to that camera (app/Trackball.h:30) and register it as an event handler with `addEventHandler` (app/Viewer.h:71). The trackball reads the window extent, so add it after the window is created.

```cpp
auto perspective = vsg::Perspective::create(30.0, 1.6, 0.1, 1000.0); // app/ProjectionMatrix.h:69
auto lookAt = vsg::LookAt::create(                                   // app/ViewMatrix.h:71
    vsg::dvec3(0.0, -10.0, 0.0), vsg::dvec3(0.0, 0.0, 0.0), vsg::dvec3(0.0, 0.0, 1.0));
auto camera = vsg::Camera::create(perspective, lookAt);              // app/Camera.h:31

auto trackball = vsg::Trackball::create(camera);                     // app/Trackball.h:30
viewer->addEventHandler(trackball);                                  // app/Viewer.h:71
```

The trackball exposes configurable key/button bindings such as `turnLeftKey` / `turnRightKey` (app/Trackball.h:87-90) and `rotateButtonMask` / `panButtonMask` / `zoomButtonMask` (app/Trackball.h:123-129), and supports key-bound viewpoints via `addKeyViewpoint` (app/Trackball.h:68).

## Loading models

Read models, images, and other serialized objects with `vsg::read` / `vsg::read_cast`. `read_cast<T>` (io/read.h:37) reads the file and `dynamic_cast`s the result to the type you want, returning a null `ref_ptr<T>` if the file is missing or of the wrong type — always check the result. An `Options` object (passed as `ref_ptr<const Options>`) carries search paths and registered readerwriters; pass `{}` for defaults.

```cpp
auto options = vsg::Options::create();                              // io/read.h:18 (Options.h, included)
if (auto model = vsg::read_cast<vsg::Node>("teapot.vsgt", options)) // io/read.h:37
{
    scene->addChild(model);                                         // nodes/Group.h:39
}
else
{
    // file missing or not a vsg::Node
}
```

For a single untyped object use `vsg::read(filename, options)` which returns a `ref_ptr<Object>` (io/read.h:24); `read` overloads also accept `std::istream&` (io/read.h:30) and raw memory (io/read.h:33).

## Event handling

Custom interaction is a `Visitor` subclass. Derive from `vsg::Inherit<vsg::Visitor, MyHandler>` so you get the `create` factory and the dispatch machinery (core/Inherit.h:27,70), then override `apply(...)` for the specific event types you care about — the `Inherit::accept` overrides route each concrete event to the matching `apply` (core/Inherit.h:70-72). The `Trackball` class is itself an example of this pattern, overriding `apply(KeyPressEvent&)`, `apply(ButtonPressEvent&)`, etc. (app/Trackball.h:38-42). Register your handler with `addEventHandler` (app/Viewer.h:71).

```cpp
#include <vsg/all.h>

class MyHandler : public vsg::Inherit<vsg::Visitor, MyHandler>      // core/Inherit.h:27
{
public:
    void apply(vsg::KeyPressEvent& keyPress) override               // cf. app/Trackball.h:38
    {
        // inspect keyPress and react
    }
};

auto handler = MyHandler::create();                                 // core/Inherit.h:35
viewer->addEventHandler(handler);                                   // app/Viewer.h:71
```

The overridable event types are declared as `virtual void apply(...)` overloads on `vsg::Visitor`: `apply(KeyPressEvent&)`, `apply(KeyReleaseEvent&)`, `apply(ButtonPressEvent&)`, `apply(ButtonReleaseEvent&)`, `apply(MoveEvent&)`, `apply(ScrollWheelEvent&)`, `apply(FrameEvent&)`, and more (core/Visitor.h:444-457); `Trackball` overrides this same set (app/Trackball.h:38-49).

## The render loop

`advanceToNextFrame()` returns `false` once the viewer is closed, so it drives the loop condition; it also polls events and advances the `FrameStamp` (app/Viewer.h:99-102). Inside the loop the canonical sequence is handle events, run update operations, record-and-submit the command buffers, then present. Each step is a `Viewer` method.

```cpp
while (viewer->advanceToNextFrame())                                // app/Viewer.h:102
{
    viewer->handleEvents();                                         // app/Viewer.h:105
    viewer->update();                                               // app/Viewer.h:135
    viewer->recordAndSubmit();                                      // app/Viewer.h:137
    viewer->present();                                              // app/Viewer.h:139
}
```

`handleEvents()` dispatches the polled `UIEvents` to every registered event handler (app/Viewer.h:104-105). `update()` runs queued update operations and animations (app/Viewer.h:135); operations are queued with `addUpdateOperation` (app/Viewer.h:85). Call `viewer->close()` (app/Viewer.h:59) from an event handler to schedule shutdown, which makes the next `advanceToNextFrame()` return false.

## Multithreaded recording

For scenes with multiple command graphs, recording can be spread across threads. After the command graphs are assigned and the viewer is compiled, call `viewer->setupThreading()` once before entering the loop; the standard loop body is unchanged because `recordAndSubmit()` cooperates with the worker threads. Call `stopThreading()` to tear the threads down.

```cpp
viewer->assignRecordAndSubmitTaskAndPresentation({commandGraph});   // app/Viewer.h:124
viewer->compile();                                                  // app/Viewer.h:107

viewer->setupThreading();                                           // app/Viewer.h:132

while (viewer->advanceToNextFrame())                                // app/Viewer.h:102
{
    viewer->handleEvents();                                         // app/Viewer.h:105
    viewer->update();                                               // app/Viewer.h:135
    viewer->recordAndSubmit();                                      // app/Viewer.h:137
    viewer->present();                                              // app/Viewer.h:139
}

viewer->stopThreading();                                            // app/Viewer.h:133
```

The viewer holds the thread pool for this: `std::list<std::thread> threads` (app/Viewer.h:130) plus the `FrameBlock`/`Barrier` synchronization primitives (app/Viewer.h:165-167). You do not need to call `stopThreading()` yourself — `Viewer::~Viewer()` invokes it during destruction (src/vsg/app/Viewer.cpp:45); call it explicitly only to tear threading down before the viewer is destroyed (app/Viewer.h:133).
