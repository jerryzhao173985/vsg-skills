# Anti-patterns

VulkanSceneGraph relies on intrusive reference counting (`vsg::Object` carries its own atomic count) and a CRTP-based factory pattern (`vsg::Inherit`). The most common mistakes by application authors come from fighting these conventions: allocating with raw `new`, holding ownership through raw or `std::shared_ptr` pointers, creating strong reference cycles, or skipping the compile step before rendering. The table below pairs each anti-pattern with the cited VSG API that replaces it. All headers below are relative to `include/vsg` and are reachable through the umbrella header `<vsg/all.h>`.

## Bad | Good | Why

| Bad | Good | Why |
| --- | --- | --- |
| `auto* g = new vsg::Group;` (raw `new`) | `auto g = vsg::Group::create();` returning `ref_ptr<Group>` (nodes/Group.h:23 via `Inherit::create` at core/Inherit.h:35-38) | `Inherit<>::create` perfect-forwards constructor args into `new Subclass(...)` and immediately wraps the result in a `ref_ptr<Subclass>`, so the object enters reference-counted ownership at birth. `Object` also routes `operator new`/`operator delete` through the `vsg::Allocator` singleton (core/Object.h:77-79); a raw `new`/`delete` bypasses nothing but still strands the object outside a `ref_ptr`. |
| Owning a node through `std::shared_ptr<vsg::Group>` or a bare `vsg::Group*` | Own through `vsg::ref_ptr<vsg::Group>` (core/ref_ptr.h:20-21) | `vsg::Object` has its own intrusive count incremented by `ref()`/decremented by `unref()` (core/Object.h:122-127), which triggers `_attemptDelete()` when the count reaches zero (core/Object.h:125,184). `ref_ptr` is the smart pointer designed for that count and is documented as half the size and faster than `std::shared_ptr` (core/ref_ptr.h:18-19). A second `std::shared_ptr` control block would double-count and double-free. |
| Calling `delete node;` on a VSG object | Let the last `ref_ptr` go out of scope / call `reset()` (core/ref_ptr.h:72-76) | `~Object` is `protected` (core/Object.h:182), so you cannot `delete` it from outside; destruction is driven solely by `unref()` reaching zero (core/Object.h:123-126). `ref_ptr::reset()` and its destructor call `unref()` for you (core/ref_ptr.h:69,74). |
| Storing a child you do not own as a raw `vsg::Node*` member and dereferencing it later | Store it as a `vsg::ref_ptr<vsg::Node>` member (nodes/Group.h:36-37) | `Group::children` is `std::vector<ref_ptr<Node>>` (nodes/Group.h:36) and `addChild` takes a `ref_ptr<Node>` (nodes/Group.h:39-42), so a child stays alive only while some `ref_ptr` holds it. A raw `Node*` does not participate in the count and dangles once the owning `ref_ptr`s drop. Even `ref_ptr`'s implicit `operator T*()` is flagged as "potentially dangerous" for exactly this reason (core/ref_ptr.h:164-165). |
| Two nodes holding each other via `ref_ptr` (parent->child AND child->parent strong refs), leaking the cycle | Make the back-reference a `vsg::observer_ptr<T>` and promote to `ref_ptr` only when accessing (core/observer_ptr.h:22-23, 172-173) | `observer_ptr` is the weak companion to `ref_ptr`, broadly equivalent to `std::weak_ptr` (core/observer_ptr.h:20-21); it does not raise the reference count, so it breaks the cycle. Convert it back safely with `ref_ptr()` (core/observer_ptr.h:173) or check `valid()` (core/observer_ptr.h:168) before use. |
| Calling `observer_ptr::get()` and dereferencing the raw pointer | Convert to `ref_ptr` first: `if (auto p = obs.ref_ptr()) p->...` (core/observer_ptr.h:172-173) | The header explicitly recommends access via `ref_ptr<>` rather than `get()`, and warns `get()` is only safe when you already know the object stays alive (core/observer_ptr.h:188-189). The conversion takes the Auxiliary mutex and re-checks the connected object before handing back a counted pointer (core/observer_ptr.h:177-186). |
| Entering the render loop right after building the scene graph, without compiling | Call `viewer->compile()` once before the loop (app/Viewer.h:107) | `Viewer::compile(ref_ptr<ResourceHints> = {})` returns a `CompileResult` (app/Viewer.h:107) that prepares Vulkan resources for the assigned command graphs; it is the step that turns the scene graph into something `recordAndSubmit()` (app/Viewer.h:137) can record. Subgraphs added later need `updateViewer(viewer, compileResult)` to wire them in (app/Viewer.h:172). |
| Calling `recordAndSubmit()`/`present()` without advancing the frame or before checking the viewer is still active | Drive each frame as `advanceToNextFrame()`, then `handleEvents()`, `update()`, `recordAndSubmit()`, `present()` (app/Viewer.h:102,105,135,137,139) | `advanceToNextFrame` checks active status (returns false when the viewer is closed), polls events, and generates the new `FrameStamp` (app/Viewer.h:99-102); skipping it records against a stale frame. `Viewer::close()` makes `active()` return false (app/Viewer.h:58-59), which is what `advanceToNextFrame` reports. |
| Copying a `Viewer` (e.g. passing it by value) | Hold and pass the `ref_ptr<Viewer>` from `Viewer::create()` (app/Viewer.h:32 via core/Inherit.h:35) | `Viewer` deletes its copy constructor and copy assignment (app/Viewer.h:37-38), so a by-value copy will not compile; it is meant to live behind a `ref_ptr`. |
| Down-casting with `dynamic_cast<vsg::StateGroup*>(node.get())` everywhere | Use `node->cast<vsg::StateGroup>()` or `vsg::cast<vsg::StateGroup>(node)` (core/Object.h:89-99, 195-205) | VSG provides its own `cast<T>()` that, when `VSG_USE_dynamic_cast` is off, uses `is_compatible(typeid(T))` + `static_cast` instead of RTTI (core/Object.h:95-99); `is_compatible` is generated per type by `Inherit` (core/Inherit.h:50). `ref_ptr` also exposes `.cast<R>()` returning a counted pointer (core/ref_ptr.h:192-193). |

## Notes

- The render-loop ordering rule lists the convenience methods that exist on `Viewer` (`advanceToNextFrame`, `handleEvents`, `update`, `recordAndSubmit`, `present`; app/Viewer.h:102-139). The exact recommended call sequence and any conditional skips are an application convention; the headers confirm each method exists and that `advanceToNextFrame` bundles the active-check, event poll, and frame-stamp advance (app/Viewer.h:99-102), but do not prescribe a single canonical loop body. Treat the listed order as the common idiom, not a header-mandated contract.
- `Inherit<>::create` is inherited by every VSG node type (e.g. `Group : Inherit<Node, Group>` at nodes/Group.h:23, `StateGroup : Inherit<Group, StateGroup>` at nodes/StateGroup.h:31, `Viewer : Inherit<Object, Viewer>` at app/Viewer.h:32), so the "use `T::create()`" rule applies uniformly across the library. The base `Object` also offers its own `create()`/`create_if()` (core/Object.h:67-75).
- `StateGroup::stateCommands` and `prototypeArrayState` are likewise `ref_ptr`-based containers (nodes/StateGroup.h:37,40,48-51); add state via `StateGroup::add(ref_ptr<StateCommand>)` (nodes/StateGroup.h:48) rather than constructing commands with raw `new`.
- No unverified items: every API named in the table was read directly in the cited headers.
